# Покупки для MAX и Telegram

Самостоятельный сервис списков покупок с одним Mini App и двумя входами:
MAX и Telegram. Оба мессенджера используют общую бизнес-логику, PostgreSQL,
каталог, рецепты и API. Отличаются только авторизация, Bot API, клавиатуры и
deep links.

## Основные механики

### Пользователь и вход

- Сервер проверяет подписанные `initData` MAX или Telegram и выдаёт HttpOnly-
  сессию.
- Внутренний пользователь отделён от внешних аккаунтов таблицей
  `user_identities`; данные разных людей не смешиваются.
- PostgreSQL — единственный источник истины. В браузере остаются только
  настройки интерфейса и короткий кэш общего каталога.

### Списки

- **Для себя** — личные списки владельца.
- **Для семьи** — общие списки участников одной семейной группы.
- **Совместно** — отдельная покупка, в которую входят по ссылке или короткому
  коду.
- Участник совместной покупки может выйти без удаления списка у остальных.
  Удаление владельцем также не должно уничтожать доступные другим участникам
  данные: владение передаётся или удаляется только связь пользователя.
- Список можно закрепить; приоритет показа: закреплённые, совместные, семейные,
  затем личные.

### Товары, блюда и совместная покупка

- Общий поиск работает по товарам, блюдам, принадлежностям и личным позициям.
- Добавленный вручную товар сохраняется в личном каталоге и доступен при
  следующем поиске.
- Повторное добавление одинакового товара с совместимой единицей суммирует
  количество, а не создаёт дубль.
- Выбор блюда и объёма/порций добавляет рассчитанный набор ингредиентов.
- Купленные позиции, отсутствие товара, количество и исполнители меняются через
  серверный API.
- Цветовые метки визуально разделяют позиции между участниками без сложной
  системы ролей.
- Открытый список обновляется у участников через авторизованный SSE-поток без
  ручной перезагрузки.

### Боты

- `/start` показывает короткое главное меню; deep link приглашения обрабатывается
  отдельно и сразу предлагает открыть подключённый список.
- `/lists` и кнопка «Мои списки» открывают списки прямо в чате.
- Товары отмечаются inline-кнопками `⬜/✅`; сообщение обновляется на месте.
- Mini App нужен для полного редактирования, каталога, блюд, кабинета и
  приглашений.

### RM Admin и рассылки

- Одна страница показывает MAX и Telegram с переключением платформы.
- Доступны статистика, поиск, полный CSV и понятные сегменты с числом
  получателей: вся аудитория, активные 7 дней, новые 7 дней, неактивные 30 дней.
- Рассылка выполняется в два действия: сначала расчёт получателей, затем явное
  подтверждение отправки.
- Кампания и каждый получатель фиксируются в PostgreSQL. Отправку с ограничением
  скорости выполняет отдельный `broadcast_worker`; перезапуск не создаёт
  повторную успешную отправку.
- Внутренний API требует `X-Admin-Token`. Токены ботов и пользователей не
  передаются в RM Admin.

## Сервисы

`compose.yaml` запускает пять независимых сервисов:

- `db` — PostgreSQL с `pgvector` и `pg_trgm`;
- `web` — FastAPI, Mini App, API, миграции и начальный seed;
- `max_bot` — MAX Long Polling;
- `telegram_bot` — Telegram Long Polling;
- `broadcast_worker` — очередь рассылок только этого проекта.

Профиль `test` поднимает отдельный `test_db` в `tmpfs`. Интеграционные тесты не
пишут пользователей, списки или кампании в рабочую локальную БД.

## Структура

```text
app/
├── api/                  # HTTP-сценарии, права, приглашения, realtime, admin API
├── bots/                 # общие чатовые списки и механики покупки
├── db/
│   ├── models/           # SQLAlchemy-модели
│   └── bootstrap_catalog/# начальный каталог, разделённый по категориям
├── platforms/
│   ├── max/              # MAX auth, Bot API и polling
│   └── telegram/         # Telegram auth, aiogram и polling
├── static/               # общий адаптивный Mini App
├── broadcast_worker.py   # точка запуска рассылок
└── main.py               # FastAPI
migrations/               # Alembic
tests/                    # unit и integration
compose.yaml
```

Исполняемый код и крупные справочники разделяются по ответственности. Исходный
файл не должен превышать 1000 строк; при приближении к 800 строкам модуль нужно
разделить заранее. Полные правила — в [`AGENTS.md`](AGENTS.md).

## Локальный запуск

```bash
cp .env.example .env
docker compose up -d --build db web max_bot telegram_bot
docker compose ps
```

`broadcast_worker` запускайте только когда проверены токены и вы действительно
готовы обрабатывать созданные кампании:

```bash
docker compose up -d --build broadcast_worker
```

- Mini App: `http://127.0.0.1:9810/`
- healthcheck: `http://127.0.0.1:9810/health`
- PostgreSQL: только `127.0.0.1:9811`

Секреты хранятся только в `.env`; шаблон без значений — в `.env.example`.

```env

ADMIN_BRIDGE_TOKEN=<случайный-длинный-секрет>

# /Users/sale/IT/RM_Systems/.env
BOT_ADMIN_SOURCES=shopping
```

`CONTENT_ADMIN_LOGIN`/`CONTENT_ADMIN_PASSWORD` и `SITE_JWT_SECRET` из корневого
`.env`. Эти данные не дублируются в проекте покупок.

Если основной VDS не имеет прямого доступа к Telegram Bot API, в production
`.env` задаётся адрес собственного доверенного HTTPS-релея:

```env
TELEGRAM_API_BASE_URL=https://trusted-relay.example/secret-path
```

Полный VPN для контейнеров не требуется: через релей идут только запросы
Telegram-бота и Telegram-рассылок. Секретный путь хранится только в production
`.env` и не коммитится.

Сайт, RM Admin и reverse proxy обновляются корневым контуром `deploy/`. Сервисы
покупок после доставки исходников запускаются из своей директории:

```bash
cd /root/RM_Systems/MAX
docker compose -f compose.yaml -f compose.prod.yaml up -d --build db web max_bot telegram_bot broadcast_worker
```

Production override подключает только `web` к существующей сети
`rm_systems_web` под уникальным alias `max_web`. БД, боты и worker остаются в
изолированной сети Compose-проекта `max`.
